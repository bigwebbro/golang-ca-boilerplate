# Golang Boilerplate - Clean Architecture

Тестовый проект на Go, демонстрирующий реализацию принципов чистой архитектуры (Clean Architecture).

## О проекте

Этот проект представляет собой boilerplate-приложение для управления пользователями и группами, построенное с соблюдением следующих архитектурных принципов:
- Чистая архитектура (Clean Architecture)
- Dependency Inversion Principle
- Event-Driven архитектура

## Стек технологий

- **Язык**: Go 1.21+
- **Web-фреймворк**: Fiber v2
- **База данных**: PostgreSQL 14
- **Контейнеризация**: Docker & Docker Compose
- **Конфигурация**: Viper

## Краткий запуск

```bash
# Первый запуск (с пересборкой контейнеров)
make docker-compose-up-build

# Последующие запуски
make docker-compose-up

# Остановка сервисов
make docker-compose-down

```

```

## Архитектура проекта

Проект построен по принципам чистой архитектуры с четким разделением на слои. 
Зависимости направлены в соотвествии с DIP от модулей низкого уровня к модулям высокого уровня, что обеспечивает высокую тестируемость и независимость бизнес-логики от инфраструктуры.

## Слои архитектуры

```

│                     APPLICATION LAYER                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │

Каждый слой это golang module, который явно определяет свои зависимости для возможной валидации и контроля в ci.

### Domain Layer (high level policy)

**Назначение**: Содержит бизнес-логику и является ядром приложения. Не имеет зависимостей от других слоев.

**Компоненты**:
- **Entities** (`domain/user/`, `domain/group/`) - бизнес-сущности с поведением
- **Value Objects** (`domain/user/type/`) - типы не имеющего жизненнего цикла для бизнес-cущностей
- **Domain Events** (`domain/user/event/`) - события происходящие в домене

**Принципы**:
- Чистые Go структуры без зависимостей от фреймворков
- Бизнес-логика инкапсулирована в методах сущностей
- События генерируются сущностями при изменениях состояния

### Application Layer (Средний слой)

**Назначение**: Оркестрация бизнес-логики и координация взаимодействий между доменом и адаптерами.

**Компоненты**:
- **Use Cases** (`application/usecase/`) - бизнес-операции и сценарии использования
- **Queries** (`application/query/`) - операции чтения данных
- **Providers** (`application/provider/`) - интерфейсы для взаимодействия с внешними системами
- **Dispatcher** (`application/dispatcher/`) - система обработки доменных событий

**Принципы**:
- Определяет интерфейсы для адаптеров (Dependency Inversion)
- Содержит оркестрацию, но не бизнес-логику
- Связывает доменные сущности с инфраструктурой


### Adapter Layer (low level modules)

**Назначение**: Обеспечивает взаимодействие приложения с внешним миром.

В проекте используется разделение адаптеров на два типа:

**Входящие адаптеры (internal/user/adapter/in/*)**
- Используют интерфейсы, определённые в более высоких слоях, инстанцируют выполнение бизнес-логики (usecases)
- Отправляют запросы (query) в БД и другие сервисы
- Через них происходит взаимодействие с текущим сервисом

**Исходящие адаптеры (internal/user/adapter)**
- Реализуют интерефейсы, определённые в более высоких слоях
- Через них происходит взаимодействие с внешними сервисами и системами
- Взаимодействие с ними происходит через обработчик интеграционных событий internal/user/adapter/event

**Query Flow (чтение данных)**:
```
in-adapter -> query -> provider (load) -> adapter
```

**UseCase Flow (выполнение бизнес-логики)**:
```
in-adapter -> usecase -> provider (load) -> domain -> event -> dispatcher -> ...adapter
```

### Event-Driven Architecture

Проект использует событийную архитектуру для развязки компонентов:

1. **Domain Events** - генерируются сущностями при изменениях состояния
2. **Dispatcher** - централизованная система обработки событий
3. **Event Handlers** - адаптеры, реагирующие на события

## Ключевые компоненты и паттерны

### Value Objects

В проекте активно используются Value Objects для представления бизнес-концепций:

- **Name, Patronymic, Surname** - фамилия, имя, отчество пользователя
- **Email** - email адрес с валидацией
- **Age** - возраст с бизнес-ограничениями

**Принципы**:
- Неизменяемость (immutability)
- Встроенная валидация при создании
- Замена по значению, а не по ссылке

### Domain Events

События доменной модели отражают важные бизнес-события:

- **UserCreated** - создание нового пользователя
- **UserActivated** - активация пользователя
- **UserDeactivated** - деактивация пользователя
- **UserGroupChanged** - изменение группы пользователя
- **MailingConsentGiven** - согласие на рассылку
- **RulesConfirmationGiven** - подтверждение правил

### Dispatcher Pattern

Централизованная система обработки доменных событий:

1. Регистрация обработчиков событий
2. Диспетчеризация событий соответствующим обработчикам
3. Развязка доменной логики от инфраструктуры
